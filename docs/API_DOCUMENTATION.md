# 🛠️ DearME API & Backend Documentation

## 🏗️ Architecture Overview
DearME uses a modern, serverless architecture designed for privacy, scalability, and ease of deployment.

- **Frontend**: React 19 + Vite + Tailwind CSS (Hosted on AWS Amplify)
- **Backend API**: FastAPI (Python 3.11)
- **Deployment**: AWS Lambda (via Mangum) + HTTP API Gateway
- **Database**: Supabase (PostgreSQL)
- **Authentication**: Supabase Auth (GoTrue)
- **Inference**: Google Gemini 2.0 Flash (for AI-generated quotes)

---

## 🔐 Security Architecture

### 🛡️ Layer 1: Access Control (JWT)
Authentication is handled by **Supabase Auth**. 
- Clients send a JWT in the `Authorization: Bearer <token>` header.
- The backend verifies this token with Supabase using the `get_current_user` dependency.
- **Service Key**: The backend uses the Supabase Service Role Key for administrative tasks, but client requests are always scoped to the authenticated user.

### 🛡️ Layer 2: Content Privacy (Encryption)
DearME implements **Zero-Knowledge-Lite** privacy. Message contents are never stored as plain text.
- **Algorithm**: Fernet Symmetric Encryption (AES-128 in CBC mode with SHA256 HMAC).
- **Library**: Python `cryptography.fernet`.
- **Key Storage**: The `ENCRYPTION_KEY` is a 32-byte Base64 string stored as a secure environment variable in AWS Lambda/Terraform.
- **Flow**: 
    1. Content is encrypted at the backend *before* hitting the database.
    2. Content is only decrypted when retrieved, and only if the `status` is `sent`.

## 🚀 API Endpoint Reference

### 1. Messages (`/api/messages`)
Manages the creation and retrieval of vault messages.

| Method | Path | Auth | Description |
| :--- | :--- | :--- | :--- |
| `POST` | `/` | ✅ | Create a new encrypted, time-locked message. |
| `GET` | `/` | ✅ | List all messages where the user is the sender OR recipient. |
| `GET` | `/{id}` | ✅ | Fetch details for a specific message. |
| `DELETE` | `/{id}` | ✅ | Permanently delete a message (Owner only). |

#### **POST Request Example**
```json
{
  "title": "To My 25-Year-Old Self",
  "recipient_email": "me@example.com",
  "content": "You made it! Keep pushing forward.",
  "scheduled_date": "2026-05-20T12:00:00Z"
}
```

---

### 2. Quotes (`/api/quote`)
Provides AI-generated inspirational content tailored to the notebook aesthetic.

| Method | Path | Auth | Description |
| :--- | :--- | :--- | :--- |
| `GET` | `/` | ❌ | Returns a daily quote generated by Gemini 2.0 Flash. |

#### **Response Schema**
```json
{
  "quote": "The echo of your past is the guide for your future.",
  "author": "DearME AI",
  "date": "2024-04-13"
}
```

## 🗄️ Database Schema
The project uses a single primary table named `messages`.

| Name | Type | Constraints |
| :--- | :--- | :--- |
| `id` | `UUID` | Primary Key, Default: `gen_random_uuid()` |
| `user_id` | `UUID` | References `auth.users`, NOT NULL |
| `title` | `TEXT` | Max length 255 |
| `recipient_email` | `TEXT` | Valid Email Format |
| `encrypted_content` | `TEXT` | Base64 Encrypted String |
| `scheduled_date` | `TIMESTAMPTZ` | NOT NULL |
| `status` | `TEXT` | Default: `scheduled`. Values: `scheduled`, `sent`, `failed` |
| `created_at` | `TIMESTAMPTZ` | Default: `now()` |

## ⚙️ Environment Configuration
The backend requires the following variables:
- `ENCRYPTION_KEY`: For Fernet cipher initialization.
- `SUPABASE_URL` & `SUPABASE_SERVICE_KEY`: For DB and Auth connection.
- `GEMINI_API_KEY`: For AI quote generation.
- `ALLOWED_ORIGINS`: For CORS management.
